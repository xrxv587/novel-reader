# 章节解析逻辑文档

## 一、概述

本文档详细描述小说阅读器中**本地 txt 书籍导入与章节解析**的完整流程。核心目标是将用户选择的 txt 文件解析为结构化的章节数据，支持后续的阅读和定位功能。

## 二、整体流程

```
用户选择 txt 文件
    ↓
复制文件到私有目录
    ↓
检测编码和 BOM
    ↓
读取全文内容
    ↓
正则匹配章节标题
    ↓
字符偏移转字节偏移
    ↓
保存章节信息到数据库
    ↓
阅读时按需加载章节内容
```

## 三、核心步骤详解

### 3.1 文件选择与复制

**文件**：[import-service.ts](file:///E:/work/novel-reader/src/utils/import-service.ts#L13-L60)

**流程**：
1. 调用 `fileUtil.chooseTxtFile()` 弹出文件选择器，用户选择 `.txt` 文件
2. 提取文件名作为书籍标题（去除 `.txt` 后缀）
3. 生成目标文件名：`${Date.now()}_${原始文件名}`（防止重名）
4. 调用 `fileUtil.copyFileToPrivate()` 将文件复制到应用私有目录 `books/` 下

**设计意图**：
- 文件复制到私有目录可防止原文件被删除或移动后导致书籍无法读取
- 添加时间戳前缀避免文件名冲突

### 3.2 编码检测

**文件**：[file.ts](file:///E:/work/novel-reader/src/utils/file.ts#L143-L159)

**流程**：
1. 读取文件前 2KB 字节内容
2. **BOM 检测**：检查前 3 字节是否为 `EF BB BF`（UTF-8 BOM）
3. **编码判断**：验证是否为合法 UTF-8，否则视为 GBK

**返回值**：
```typescript
{
  encoding: 'utf-8' | 'gbk',  // 文件编码
  hasBom: boolean              // 是否包含 UTF-8 BOM
}
```

**关键处理**：
- 若存在 BOM，后续读取时需跳过前 3 字节（`startOffset = 3`）
- 有效文件大小需减去 3 字节（`effectiveSize = fileSize - 3`）

### 3.3 章节解析

**文件**：[chapter.ts](file:///E:/work/novel-reader/src/utils/chapter.ts#L10-L46)

**章节匹配正则**：
```typescript
/^\s*(第[一二三四五六七八九十百千零\d]+[章节回卷部][\s、．。：:]*.*)$/gm
```

**匹配规则**：
| 组成部分 | 说明 | 示例 |
|---------|------|------|
| `第` | 章节标题前缀 | 第 |
| `[一二三四五六七八九十百千零\d]+` | 章节编号（中文数字或阿拉伯数字） | 一、二、三、1、2、3 |
| `[章节回卷部]` | 章节类型后缀 | 章、节、回、卷、部 |
| `[\s、．。：:]*` | 分隔符（空格、顿号、句号、冒号等） | 空、、、。、：、: |
| `.*` | 章节副标题（可选） | 返乡、旧屋、神秘的森林 |

**解析逻辑**：
1. 使用正则全局匹配所有章节标题
2. 每匹配到一个章节标题：
   - 更新上一个章节的 `endIndex`（设为当前匹配位置）
   - 计算上一个章节的 `wordCount`
   - 创建新章节对象，`startIndex` 设为当前匹配位置
3. 最后一个章节的 `endIndex` 设为内容总长度

**Fallback 处理**：
- 若未匹配到任何章节标题，调用 `createSingleChapter()` 将整本书作为一个章节
- 标题取文件第一行前 30 个字符

**章节数据结构**：
```typescript
interface ChapterInfo {
  title: string;       // 章节标题
  startIndex: number;  // 字符级起始位置
  endIndex: number;    // 字符级结束位置（下一章标题位置）
  wordCount: number;   // 字符数
}
```

### 3.4 字符偏移转字节偏移

**文件**：[import-service.ts](file:///E:/work/novel-reader/src/utils/import-service.ts#L62-L81)

**问题背景**：
- `parseChapters()` 返回的 `startIndex`/`endIndex` 是**字符索引**（基于内存中的字符串）
- `file.slice(start, end)` 读取文件时使用的是**字节偏移量**
- UTF-8 编码下，中文字符占 3 字节，直接混用会导致严重偏移

**转换逻辑**：
```typescript
function getByteOffset(content: string, charIndex: number, encoding: string): number {
  // GBK/GB2312 每个字符固定 2 字节
  if (encoding.toLowerCase() === 'gbk' || encoding.toLowerCase() === 'gb2312') {
    return charIndex * 2;
  }
  
  // UTF-8 变长编码
  let byteOffset = 0;
  for (let i = 0; i < charIndex && i < content.length; i++) {
    const charCode = content.charCodeAt(i);
    if (charCode < 0x80) {
      byteOffset += 1;  // ASCII 字符
    } else if (charCode < 0x800) {
      byteOffset += 2;  // 双字节字符
    } else if (charCode < 0x10000) {
      byteOffset += 3;  // 三字节字符（中文）
    } else {
      byteOffset += 4;  // 四字节字符（罕见）
    }
  }
  return byteOffset;
}
```

**章节偏移修正**：
```typescript
const byteChapters = chapters.map(ch => ({
  ...ch,
  // 起始偏移：BOM 偏移 + 字符转字节偏移
  startIndex: startOffset + getByteOffset(content, ch.startIndex, encoding),
  // 结束偏移：BOM 偏移 + (endIndex - 1 字符转字节偏移)
  // 减 1 是因为 parseChapters 返回的 endIndex 是下一章标题位置，需取当前章最后一个字符
  endIndex: startOffset + getByteOffset(content, ch.endIndex - 1, encoding)
}));
```

### 3.5 章节数据保存

**文件**：[chapter-db.ts](file:///E:/work/novel-reader/src/utils/chapter-db.ts#L16-L29)

**保存逻辑**：
1. 删除该书已有的章节数据（`DELETE FROM chapters WHERE book_id = ?`）
2. 逐条插入章节记录：
   - `book_id` - 书籍 ID
   - `chapter_index` - 章节顺序索引（从 0 开始）
   - `title` - 章节标题
   - `start_offset` - 字节级起始偏移
   - `end_offset` - 字节级结束偏移
   - `word_count` - 字符数

### 3.6 章节内容加载

**文件**：[import-service.ts](file:///E:/work/novel-reader/src/utils/import-service.ts#L106-L134)

**加载流程**：
1. 根据 `bookId` 和 `chapterIndex` 查询章节记录
2. **缓存优先**：若 `chapter.content` 已缓存，直接解析 JSON 返回
3. **文件读取**：调用 `fileUtil.readFileWithEncoding()` 读取指定字节范围
4. **段落分割**：调用 `splitIntoParagraphs()` 按换行符分割段落
5. **缓存写入**：将段落数组 JSON 序列化后存入 `chapter.content` 字段

**段落分割逻辑**（[chapter.ts](file:///E:/work/novel-reader/src/utils/chapter.ts#L63-L68)）：
```typescript
export function splitIntoParagraphs(content: string): string[] {
  return content
    .split(/\n+/)           // 按一个或多个换行符分割
    .map(p => p.trim())     // 去除首尾空白
    .filter(p => p.length > 0);  // 过滤空段落
}
```

## 四、关键技术点

### 4.1 APP-PLUS 文件切片是闭区间

**问题**：`plus.io.File.slice(start, end)` 是**闭区间**（start 和 end 都包含），而标准 Web API 是半开区间 `[start, end)`。

**解决方案**：在 `parseBookChapters` 中，将 `endIndex` 转换为当前章节最后一个字符的字节偏移量（`ch.endIndex - 1`），确保读取时不会包含下一章标题的第一个字符。

### 4.2 UTF-8 BOM 处理

**问题**：部分 txt 文件开头包含 UTF-8 BOM（`EF BB BF`），直接读取会显示为乱码。

**解决方案**：
1. 检测阶段识别 BOM 存在
2. 计算有效文件大小（减去 3 字节）
3. 读取时跳过前 3 字节
4. 章节偏移量加上 BOM 偏移

### 4.3 字符偏移与字节偏移转换

**问题**：章节解析基于字符索引，但文件读取使用字节偏移。

**解决方案**：根据文件编码，逐字符计算字节偏移量：
- GBK：每个字符固定 2 字节
- UTF-8：ASCII 1 字节，中文 3 字节

### 4.4 章节缓存策略

**设计**：已阅读过的章节内容（段落数组）JSON 序列化后存入 `chapters.content` 字段。

**优点**：
- 避免重复读取文件，提升阅读流畅度
- 减少文件 I/O 操作，降低性能开销

## 六、注意事项

1. **重新导入**：章节信息保存在数据库中，修改解析逻辑后需重新导入书籍才能生效
2. **缓存清理**：章节内容缓存在 `chapters.content` 字段，若需重新解析，需清空此字段
3. **编码兼容性**：仅支持 UTF-8 和 GBK 两种编码，其他编码可能导致乱码
4. **文件大小**：当前实现为一次性读取全文，超大文件（>100MB）可能导致内存问题，未来可优化为流式分块读取
