# 小说阅读器技术方案

## 一、项目概述

基于 **uni-app** 框架开发的跨平台小说阅读器 App，支持 iOS 和 Android 双端。核心能力包括本地 txt 小说导入、在线书源搜索、沉浸式阅读体验、排版美化和书签管理。

### 1.1 技术选型结论

| 维度 | 选型 | 说明 |
|------|------|------|
| 框架 | uni-app (Vue 3 + Vite) | 一套代码编译到 iOS / Android，生态成熟 |
| UI 组件库 | uView Plus 或 uni-ui | 适配 uni-app 的跨端组件库 |
| 状态管理 | Pinia | Vue 3 官方推荐，轻量易用 |
| 本地存储 | plus.sqlite + uni.setStorage | SQLite 存结构化数据，storage 存小配置 |
| 文件选择 | plus.io.chooseFile（或原生插件） | App 端选择本地 txt 文件 |
| 文件读取 | plus.io FileReader + file.slice | 支持分块读取大文件 |
| 网络请求 | uni.request | 书源搜索和内容抓取，原生端无跨域限制 |
| HTML 解析 | cheerio@0.22.0 | 纯 JS 实现，jQuery 风格 API，App 端可用 |
| 编码检测/转码 | jschardet + iconv-lite | 纯 JS 库，支持 GBK/UTF-8 等编码 |
| 架构模式 | 纯前端（无服务端） | 所有逻辑在 App 内完成 |

> **可行性验证结论**：以上所有依赖均已验证可在 uni-app App 端（iOS/Android）正常运行，无不可逾越的技术障碍。

### 1.2 为什么选纯前端架构

- **开发成本低**：不需要后端服务器，省去服务器运维和域名等成本
- **上线快**：纯客户端开发，不用联调接口，一个人就能搞定
- **隐私友好**：用户数据全部存在本地，不存在数据泄露风险
- **uni-app 原生端无跨域限制**：编译到 iOS/Android 后，`uni.request` 可以直接请求任何网站，不受浏览器跨域限制
- **可平滑升级**：书源逻辑封装成独立模块，未来需要后端时只需替换实现

### 1.3 为什么选 uni-app

- **跨端能力**：一套代码编译 iOS / Android，后期还能扩展到小程序/H5
- **生态成熟**：插件市场丰富，文件选择、本地存储、原生插件都有现成方案
- **性能够用**：阅读页核心用 nvue 或自定义渲染，大文件采用虚拟滚动方案
- **开发效率**：Vue 语法上手快，热更新方便调试

## 三、核心模块设计

### 3.1 书架模块

**数据结构 — 书籍表 (books)**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 书籍唯一标识 |
| title | TEXT | 书名 |
| author | TEXT | 作者（可空） |
| source_type | TEXT | `local` 本地导入 / `online` 在线书源 |
| file_path | TEXT | 本地文件路径（本地书用） |
| source_url | TEXT | 书源地址（在线书用） |
| total_chapters | INTEGER | 总章节数 |
| last_read_chapter | INTEGER | 最后阅读章节索引 |
| last_read_position | INTEGER | 最后阅读位置（字符偏移或滚动位置） |
| cover_image | TEXT | 封面图路径（可空） |
| created_at | TIMESTAMP | 添加时间 |
| updated_at | TIMESTAMP | 最后阅读时间 |

**书架页逻辑**

1. 进入页面从 SQLite 查询书籍列表，按 `updated_at` 倒序排列
2. 每本书展示：封面、书名、阅读进度（百分比）
3. 点击书籍 → 跳转阅读页，带上 `bookId`
4. 阅读页读取 `last_read_chapter` 和 `last_read_position`，自动跳转到上次阅读位置
5. 左滑删除 → 确认后删除书籍记录和本地文件

### 3.2 文件导入模块

**导入流程**

```
用户点击"导入书本"
    ↓
调用 plus.io.chooseFile 选择 .txt 文件
    ↓
将文件复制到 App 私有目录（防止原文件被删）
    ↓
解析文件名作为书名（可编辑）
    ↓
读取文件前 2KB，用 jschardet 检测编码
    ↓
生成书籍元数据，写入 books 表
    ↓
异步解析章节（后台执行，不阻塞UI）
    ↓
返回书架，刷新列表
```

**文件选择方案**

`uni.chooseFile` 在 App 端支持不完善，推荐使用以下方案：

| 方案 | 说明 | 推荐度 |
|------|------|--------|
| `plus.io.chooseFile` | 5+ 原生 API，iOS/Android 均支持 | ⭐⭐⭐ |
| DCloud 插件市场原生文件选择器 | 如 `dh-choose-file`，UI 更友好 | ⭐⭐⭐⭐ |
| `<input type="file">` | 简单但体验一般，不推荐 | ⭐ |

推荐优先用 `plus.io.chooseFile`，有更高 UI 要求再换原生插件。

**编码识别与转码**

- 用 `jschardet` 检测文件编码（UTF-8 / GBK / GB2312 等）
- 检测到非 UTF-8 编码时，用 `iconv-lite` 转码
- 编码信息记录在书籍表中，阅读时使用对应编码读取
- jschardet 和 iconv-lite 都是纯 JS 实现，App 端可直接使用

**大文件处理**

- 导入时**不读取全文**，只读取文件头部 2KB 检测编码和提取信息
- 文件大小和总字数在后台异步计算，不阻塞 UI
- 章节解析在后台异步执行，解析完一章存一章
- 书籍内容在阅读时按需读取（见阅读引擎章节）

**注意事项**

- `plus.io` 的 `file.slice(start, end)` 是**闭区间**（包含 end 位置的字节），和 Web 标准的半开区间不同，读取时需要 `end - 1` 修正
- 大文件建议分块读取，单块 50KB ~ 100KB 为宜

### 3.3 阅读引擎（核心难点）

这是整个项目最关键的部分，直接决定体验好坏。

#### 3.3.1 本地 txt 书籍的章节化处理

**问题**：txt 文件没有章节结构，整本书是一整块文本。

**解决方案**：导入时异步解析章节，按"第X章"等正则切分。

```javascript
// 章节匹配正则（覆盖常见写法）
const chapterRegex = /^\s*(第[一二三四五六七八九十百千零\d]+[章节回卷部][\s、．。：:]*.*)$/gm;

function parseChapters(content) {
  const chapters = [];
  let match;
  let lastIndex = 0;
  
  while ((match = chapterRegex.exec(content)) !== null) {
    if (chapters.length > 0) {
      // 补上一章的结尾位置
      chapters[chapters.length - 1].endIndex = match.index;
    }
    chapters.push({
      title: match[1].trim(),
      startIndex: match.index,
      endIndex: content.length // 先默认，下一章会修正
    });
    lastIndex = match.index;
  }
  
  // 如果没匹配到章节，按固定字数（约5000字）分章
  if (chapters.length === 0) {
    const chunkSize = 5000;
    for (let i = 0; i < content.length; i += chunkSize) {
      chapters.push({
        title: `第${Math.floor(i / chunkSize) + 1}章`,
        startIndex: i,
        endIndex: Math.min(i + chunkSize, content.length)
      });
    }
  }
  
  return chapters;
}
```

**章节索引存储**：解析完的章节信息（标题、起始偏移、结束偏移）存入 SQLite 的 `chapters` 表，和 `bookId` 关联。

#### 3.3.2 大文件按需加载（虚拟阅读）

**问题**：一本小说几 MB 到几十 MB，一次性加载到内存会卡顿甚至崩溃。

**解决方案**：基于文件指针的**按需分段读取**。

```
┌─────────────────────────────────────────────┐
│               完整 txt 文件                   │
│  ┌────┬────┬────┬────┬────┬────┬────┐      │
│  │ Ch1│ Ch2│ Ch3│ Ch4│ Ch5│ Ch6│ Ch7│ ...  │
│  └────┴────┴────┴────┴────┴────┴────┘      │
│         ↑    ↑    ↑                          │
│         └────┴────┘ ← 当前只加载这3章        │
│       上一章  当前  下一章                    │
└─────────────────────────────────────────────┘
```

**实现思路**：

1. 阅读时，只加载**当前章节 + 前后各一章**到内存
2. 翻到下一章时，加载下一章内容，卸载上上一章
3. 内存中始终保持 3 章左右的内容，保证翻页流畅

**App 端文件读取**（使用 plus.io）：

```javascript
// 读取文件指定范围 [start, end) —— 语义上是左闭右开
// 注意：plus.io 的 file.slice 是闭区间（包含 end），所以内部需要 end - 1
function readFileRange(filePath, start, end, encoding = 'utf-8') {
  return new Promise((resolve, reject) => {
    plus.io.requestFileSystem(plus.io.PRIVATE_DOC, (fs) => {
      fs.root.getFile(filePath, { create: false }, (fileEntry) => {
        fileEntry.file((file) => {
          const reader = new plus.io.FileReader();
          reader.onload = (e) => resolve(e.target.result);
          reader.onerror = reject;
          
          // 关键：plus.io 的 slice 是闭区间 [start, end]，都包含
          // 为了和标准 Web API 语义一致（左闭右开），这里 end - 1
          const blob = file.slice(start, end - 1);
          reader.readAsText(blob, encoding);
        }, reject);
      }, reject);
    }, reject);
  });
}

// GBK 编码文件读取示例
async function readGBKChapter(filePath, start, end) {
  // 以二进制读取
  const arrayBuffer = await readFileAsArrayBuffer(filePath, start, end);
  // 用 iconv-lite 转码
  const text = iconvLite.decode(Buffer.from(arrayBuffer), 'gbk');
  return text;
}
```

> **重要坑点**：plus.io 的 `file.slice(start, end)` 是**闭区间**（start 和 end 位置的字节都包含），而标准 Web API 是半开区间 `[start, end)`。如果直接用会导致每次多读 1 个字节，拼接时出现重复字符。必须用 `end - 1` 修正。
>
> **适用范围**：本方案目标是 iOS / Android 原生 App，使用 plus.io 原生能力，不受小程序端限制。

#### 3.5.3 在线小说到阅读器的数据流

这是最关键的理解点：**不管是本地 txt 还是在线网页，最终喂给阅读页的都是「段落数组」**。

```
本地 txt 路径：
  txt文件 → 分块读取 → 文本 → 按段落分割 → 段落数组 → 阅读页渲染

在线小说路径：
  网页URL → uni.request → HTML字符串 → cheerio提取正文 → 按<br>/<p>分割 → 段落数组 → 阅读页渲染
```

**统一的数据格式**：

```javascript
// 阅读页的输入数据格式 —— 段落数组
const paragraphs = [
  '第一章 风起',
  '夜色如墨，寒风呼啸...',
  '他站在山巅，遥望远方...',
  // ...
];
```

阅读页组件完全不关心数据来源，只负责渲染段落数组 + 处理滚动/翻页/排版。本地书和在线书共用同一套阅读引擎。

**在线章节正文提取流程**：

```javascript
import cheerio from 'cheerio';
import iconv from 'iconv-lite';

async function fetchChapterContent(source, chapterUrl) {
  // 1. 请求章节页面
  const res = await uni.request({
    url: chapterUrl,
    method: 'GET',
    timeout: 8000,
    // 以 arraybuffer 方式接收，方便转码
    responseType: source.charset === 'GBK' ? 'arraybuffer' : 'text'
  });
  
  let html = res.data;
  
  // 2. GBK 转码
  if (source.charset === 'GBK') {
    html = iconv.decode(Buffer.from(res.data), 'gbk');
  }
  
  // 3. cheerio 加载解析
  const $ = cheerio.load(html);
  
  // 4. 移除广告元素
  if (source.content.removeSelectors) {
    source.content.removeSelectors.forEach(sel => {
      $(sel).remove();
    });
  }
  
  // 5. 提取正文纯文本
  const contentText = $(source.content.contentSelector).text();
  
  // 6. 按行分割成段落数组
  const paragraphs = contentText
    .split(/\n+/)
    .map(p => p.trim())
    .filter(p => p.length > 0);
  
  return paragraphs;
}
```

**缓存策略**：

- 章节目录：抓取后存入 SQLite，下次打开直接读缓存
- 章节正文：阅读过的章节存入 SQLite（段落数组 JSON），下次离线也能看
- 缓存有效期：可配置（默认 7 天），过期后自动重新抓取

#### 3.5.4 搜索流程（纯前端）

```
用户输入关键词
    ↓

    ↓
并发调用每个书源的搜索请求（uni.request）
    ↓
每个书源独立：请求 → 解析HTML → 提取结果
    ↓
合并所有书源的结果，按书源分组
    ↓
展示搜索结果列表
    ↓
用户点击某本书 → 抓取章节目录 → 加入书架
```

**并发搜索代码示例（cheerio 版本）：**

```javascript
import cheerio from 'cheerio';
import iconv from 'iconv-lite';

async function searchAllSources(keyword) {
  const sources = await getEnabledSources(); // 从数据库读取
  
  const promises = sources.map(source => 
    searchOneSource(source, keyword)
      .then(results => ({ source, results }))
      .catch(err => ({ source, results: [], error: err }))
  );
  
  const results = await Promise.all(promises);
  return results.filter(r => r.results.length > 0);
}

async function searchOneSource(source, keyword) {
  const url = source.baseUrl + source.search.url.replace('{keyword}', encodeURIComponent(keyword));
  
  const res = await uni.request({
    url,
    method: source.search.method,
    timeout: 8000,
    responseType: source.charset === 'GBK' ? 'arraybuffer' : 'text'
  });
  
  let html = res.data;
  // GBK 编码转码
  if (source.charset === 'GBK') {
    html = iconv.decode(Buffer.from(res.data), 'gbk');
  }
  
  // cheerio 解析
  const $ = cheerio.load(html);
  const items = $(source.search.resultList);
  
  const results = [];
  items.each((i, el) => {
    const $el = $(el);
    const title = $el.find(source.search.resultTitle).text().trim();
    const author = $el.find(source.search.resultAuthor).text().trim();
    const link = $el.find(source.search.resultLink).attr('href') || '';
    
    if (title) {
      results.push({
        title,
        author,
        link: resolveUrl(source.baseUrl, link),
        sourceId: source.id,
        sourceName: source.name
      });
    }
  });
  
  return results;
}
```

#### 3.5.5 章节目录抓取

用户选择某本书后，抓取章节目录并缓存：

```javascript
async function fetchChapters(source, bookUrl) {
  const res = await uni.request({
    url: bookUrl,
    timeout: 8000,
    responseType: source.charset === 'GBK' ? 'arraybuffer' : 'text'
  });
  
  let html = res.data;
  if (source.charset === 'GBK') {
    html = iconv.decode(Buffer.from(res.data), 'gbk');
  }
  
  const $ = cheerio.load(html);
  const items = $(source.catalog.chapterList);
  
  const chapters = [];
  items.each((idx, el) => {
    const $el = $(el);
    const $link = $el.find(source.catalog.chapterLink);
    const title = $link.text().trim();
    const href = $link.attr('href') || '';
    
    if (title) {
      chapters.push({
        title,
        url: resolveUrl(source.baseUrl, href),
        index: idx
      });
    }
  });
  
  return chapters;
}
```

抓取到的章节目录存入 SQLite 的 `chapters` 表（在线书籍关联 `book_id`），下次打开直接读缓存，不用重复请求。

#### 3.5.6 正文抓取与清洗

阅读在线章节时，请求正文页面并提取内容（前面数据流章节已有完整代码，这里补充清洗逻辑）：

**内容清洗要点：**

- 移除广告、推荐阅读等干扰元素
- 去除多余空行和空格
- 过滤掉"本章未完"、"请继续阅读"等提示文字
- 按段落分割，统一格式

**正文缓存策略**：已阅读过的章节正文缓存在 SQLite（段落数组转 JSON），下次直接读取，不用重新请求。缓存可设置过期时间（默认 7 天）。

#### 3.5.7 书源验证

用户添加书源时，自动验证配置是否有效：

```
用户填写书源配置
    ↓
测试1：发送搜索请求，验证能否获取到 ≥1 条结果
    ↓
测试2：取第一条结果，抓取章节目录，验证能否获取到章节
    ↓
测试3：取第一章，抓取正文，验证能否提取到内容
    ↓
全部通过 → 保存书源
有失败 → 提示用户哪一步失败了
```

#### 3.5.8 预置书源

MVP 版本内置 3-5 个经过验证的稳定书源，打包在 App 内。首次启动时自动写入数据库。

**书源导入导出**：支持 JSON 格式，用户可以分享书源配置，也可以从网上导入别人分享的书源。

#### 3.5.9 纯前端方案的注意事项

| 问题 | 解决方案 |
|------|----------|
| 部分网站有反爬 | 设置合理的 User-Agent，请求间隔控制 |
| GBK 编码网站 | 用 iconv-lite 前端版转码 |
| 请求失败 | 超时 + 重试机制（最多重试2次） |
| 书源挂了 | 多书源并发，一个挂了不影响其他；用户可自行添加新源 |
| 未来要加后端 | 书源逻辑封装成独立模块，以后替换实现即可 |

## 六、关键技术难点与解决方案

> **可行性总览**：本方案所有核心技术依赖均已验证可在 uni-app App 端（iOS/Android）正常运行。以下是已知的坑点和对应解决方案。

### 已知坑点速查表

| 坑点 | 影响范围 | 严重程度 | 解决方案 |
|------|----------|----------|----------|
| plus.io file.slice 是闭区间 | 本地文件分块读取 | 高 | 读取时 end - 1 修正 |
| uni.chooseFile App 端不完善 | 文件导入 | 中 | 用 plus.io.chooseFile 或原生插件 |
| cheerio 高版本不可用 | HTML 解析 | 高 | 锁定 0.22.0 版本 |
| GBK 编码网站/文件乱码 | 本地书+在线书 | 中 | jschardet 检测 + iconv-lite 转码 |
| 部分网站反爬 | 在线书源 | 中 | 多书源分散 + 请求间隔控制 + UA 伪装 |

### 难点 1：大文件加载性能

**问题**：几十 MB 的 txt 文件直接读入内存会导致 App 卡顿甚至 OOM 崩溃。

**方案**：
- 采用**文件指针偏移 + 分段读取**，只加载当前章节前后的内容
- 章节信息预解析存入数据库，阅读时根据 offset 直接 seek
- 文本内容做分段缓存，避免重复读取文件

### 难点 2：翻页流畅度

**问题**：章节切换时如果同步读取文件会有明显卡顿感。

**方案**：
- 预加载下一章：当前章读到 80% 时，后台异步加载下一章
- 上一章内容保留在内存中，往回翻直接显示
- 使用 CSS `transform: translate3d` 开启硬件加速

### 难点 3：书源稳定性（纯前端场景）

**问题**：第三方书源网站经常改版或挂掉，纯前端场景下无法集中更新，需要发版才能改书源配置。

**方案**：
- 书源可配置化，用户可自行添加/编辑书源（不用发版）
- 搜索时多源并发，一个挂了其他还能用
- 支持书源导入导出（JSON 格式），用户社区可以互相分享
- 预置书源打包在 App 内，首次启动写入数据库；后续更新通过发版或用户手动导入

### 难点 4：cheerio 版本兼容性

**问题**：高版本 cheerio（1.0+）依赖较多 Node.js API，在 uni-app App 端的 JS 引擎里跑不起来。

**方案**：
- 锁定使用 `cheerio@0.22.0` 版本，纯 JS 实现，App 端可直接运行
- 0.22.0 版本 API 完整（CSS 选择器、DOM 操作等），足够小说解析使用
- `package.json` 中固定版本号：`"cheerio": "0.22.0"`（不要加 ^）
- 不支持动态渲染的页面（JS 渲染），MVP 阶段只支持静态 HTML 页面的书源

### 难点 5：部分网站反爬

**问题**：纯前端请求容易被识别和封禁，尤其是高频请求时。

**方案**：
- 设置合理的 User-Agent（模拟手机浏览器）
- 控制请求频率，同一书源两次请求间隔 ≥ 1 秒
- 多书源分散请求，降低单站压力
- 正文内容缓存到本地，已读章节不重复请求
- 如果反爬太严重，留好后续加服务端代理的接口扩展位

### 难点 6：GBK 编码的 txt 文件和网站

**问题**：很多老小说 txt 和部分小说网站是 GBK 编码，直接用 UTF-8 读取会乱码。

**方案**：
- 本地 txt：导入时用 `jschardet` 检测编码，检测到 GBK/GB2312 时记录编码信息
- 在线书源：书源配置里加 `charset` 字段，请求后用 `iconv-lite` 转码
- 转码在前端完成，`iconv-lite` 有浏览器兼容版本
- 转码后的内容不重新存储（省空间），读取/展示时实时转码

### 难点 7：阅读位置精确还原

**问题**：退出时记录的滚动位置，下次打开可能因为排版设置变化而对不上。

**方案**：
- 存储位置用**字符偏移量**而不是像素偏移
- 打开时根据字符偏移定位到具体段落，再滚动到该段落顶部
- 排版设置变化时也能准确定位到同一位置

## 九、总结

这份方案的核心思路是：

1. **用 uni-app 做跨端**，一套代码跑 iOS 和 Android，开发效率高
2. **本地阅读是核心**，先把 txt 导入和阅读体验做好，再做在线书源
3. **大文件用按需加载**，按章节分段读取，内存中只保留 3 章左右
4. **书源做成配置化**，用户可自行添加，解决第三方源不稳定的问题
5. **分阶段开发**，先做 MVP 验证核心体验，再逐步加功能

建议从第一阶段开始，先把架子搭起来，然后用 2-3 周把本地阅读核心做出来，自己用着觉得舒服了再往下做。
